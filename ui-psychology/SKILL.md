# 🧠 UI Psychology Skill

> 用心理學框架驅動每一個設計決策——  
> 不只讓介面「好看」，而是讓用戶「忍不住行動」。

**作者**：Jason Tsai  
**版本**：v2.0  
**參考文件**：[principles.md](./references/principles.md) · [userflow.md](./references/userflow.md) · [patterns.md](./references/patterns.md) · [audit-checklist.md](./references/audit-checklist.md)

## 為什麼需要這個 Skill？

大多數設計師知道「要讓介面直觀」，卻很少能說清楚：
- 為什麼這個 CTA 沒有人點？
- 為什麼用戶填到一半就放棄表單？
- 為什麼定價頁的轉換率只有 1%？

這個 Skill 把 20+ 個心理學研究結論轉化為可執行的設計決策，讓你在每一個設計選擇背後，都能說出「**因為用戶的大腦會這樣反應，所以我這樣設計**」。

## 核心框架：三層心理學模型

所有模式都從這三層出發思考，**缺一不可**。

### 👁️ Layer 1 — 感知層（Perception）
> 用戶「看到」什麼？在意識介入之前，大腦自動捕捉了什麼？

| 原則 | 設計意義 |
|------|---------|
| 前注意特徵 | 顏色、大小、動態在 250ms 內決定注意力分配 |
| 視覺層級 | F-pattern（內容頁）/ Z-pattern（Landing Page）引導視線動線 |
| 色彩心理 | 藍=信任、紅=緊迫、綠=安全、橘=活力 |
| 空白 | 空白越多，感知品質越高端 |
| 格式塔原則 | 靠近的元素被視為一組；相似的元素被視為同類 |

### 🧠 Layer 2 — 認知層（Cognition）
> 用戶「理解」什麼？需要花多少腦力？

| 原則 | 設計意義 |
|------|---------|
| Cognitive Load | 每個額外選項、步驟都消耗決策能量 |
| Hick's Law | 選項數 ↑ → 決策時間呈對數增長（選項 ≤ 7）|
| Miller's Law | 工作記憶上限 7±2 個項目 |
| 漸進式呈現 | 先呈現核心，其他按需展開 |
| 心智模型匹配 | 設計要符合用戶的既有認知習慣 |

### 🎯 Layer 3 — 行為層（Behavior）
> 用戶「做」什麼？什麼驅動或阻礙行動？

| 原則 | 設計意義 |
|------|---------|
| 損失規避 | 損失的痛苦是收穫快樂的 2 倍（Kahneman）|
| 錨定效應 | 第一個數字決定後續判斷基準 |
| 社會認同 | 「10,000 人使用」降低決策風險感 |
| Zeigarnik Effect | 未完成的任務更佔據記憶，形成拉力 |
| 互惠原則 | 先給價值，再請求行動 |
| 承諾一致性 | 小承諾 → 大行動（Foot-in-the-door）|
| 峰終定律 | 記憶由「最高峰」和「最後一刻」決定 |

## MODE A：ANALYZE — 分析模式

**觸發**：用戶描述現有設計，想知道為什麼有效或無效

### 執行步驟
1. 對照三層模型，逐層分析問題
2. 參考 [`references/patterns.md`](./references/patterns.md) 對應 UI Pattern
3. 輸出以下結構

### 輸出格式

```markdown
## 🔍 UI 心理學分析報告｜[頁面/功能名稱]

> 分析日期：YYYY-MM-DD｜版本：[描述]

### 📊 三層評分總覽
| 維度 | 分數（/10）| 最關鍵問題 |
|------|:--------:|---------|
| 👁️ 感知層（Perception） | X | [一句話] |
| 🧠 認知層（Cognition） | X | [一句話] |
| 🎯 行為層（Behavior） | X | [一句話] |
| **整體轉換潛力** | **X** | [一句話] |

### 👁️ 感知層分析
[視覺層級 / 色彩 / 空白 / 字體 的現狀與問題，各 2-3 句]

### 🧠 認知層分析
[認知負荷 / 資訊架構 / 心智模型匹配度 的問題，各 2-3 句]

### 🎯 行為層分析
[CTA / 信任信號 / 摩擦點 / 損失規避 / 社會認同 的現狀，各 2-3 句]

### ⚡ 優先修改清單（按 ROI 排序）
| 優先級 | 問題 | 心理學原因 | 具體改法 | 預期效果 |
|:-----:|------|-----------|---------|---------|
| 🔴 P0 | [問題] | [原則] | 把 XX 改成 XX | 轉換率 +X% |
| 🟡 P1 | [問題] | [原則] | 把 XX 改成 XX | — |
| 🟢 P2 | [問題] | [原則] | 把 XX 改成 XX | — |

---
*參考原則詳見 [principles.md](./references/principles.md)*
```

## MODE U：USERFLOW — User Flow 設計模式

**觸發**：用戶要設計或優化用戶旅程（註冊、購買、Onboarding、引導流程等）

### 前置釐清（信息不足時先問這幾個問題）
- 產品類型？（SaaS / 電商 / App / 內容平台）
- 流程**起點**？（用戶從哪裡進來）
- 流程**終點目標**？（購買 / 完成設定 / 升級 / 分享）
- 用戶的**核心障礙**？（不信任 / 認知負荷高 / 不知道價值 / 步驟太多）
- 是優化現有流程，還是從零設計？

### 執行步驟
1. 讀取 [`references/userflow.md`](./references/userflow.md) — 取得 PACE 框架與各 Flow 類型策略
2. 用「心理旅程地圖」分析每步驟的用戶心理狀態
3. 依輸出格式產出：**主 Markdown 文件** + 輔助互動原型 + 設計規格連結

### 輸出格式

#### 主文件（Markdown）— 所有內容的入口

```markdown
## 🧭 [流程名稱] — 心理學 User Flow

> 版本：v1.0｜產品類型：XX｜目標行動：XX｜設計者：XX

### 📌 設計摘要
- **流程目標**：[一句話]
- **核心心理策略**：[用了哪 2-3 個主要原則]
- **預期效果**：[完成率 / 轉換率目標]

### 🗺️ Flow 總覽
[步驟 1] → [步驟 2] → [步驟 3] → ... → [完成]

→ **互動 Flow 圖**：[查看 HTML/React 原型](#prototype)
→ **設計規格文件**：[design-spec.md](./design-spec.md)

---

### 步驟 N：[步驟名稱]

**用戶心理狀態**：[此刻的情緒 / 疑慮 / 動機，一句話]

**設計規格**：→ [§N 詳細規格](./design-spec.md#step-N)

#### 摩擦點 × 解法
| 摩擦點 | 心理學原因 | 解法 |
|--------|-----------|-----|
| [摩擦] | [原則] | [具體改法] |

#### 分支邏輯
- ✅ 完成 → 步驟 N+1
- ❌ 放棄 → [挽回策略：Email / Popup / 降低門檻]
- ⚠️ 錯誤 → [即時錯誤處理]

---
[重複步驟結構至流程結束]

| 步驟 | 用戶內心獨白 | 情緒 | Pain | Anxiety | Confidence | 設計應對 |
```

#### 輔助文件規範

**互動原型**（HTML/React，從主文件 `#prototype` 錨點連結）：
- 可點擊的流程圖，每個節點展開顯示元件清單
- 顏色標記：🟢 順暢路徑 / 🔴 流失風險點 / 🟡 分支決策
- 在環境支援時主動生成；無法生成時以 Mermaid 流程圖替代

**設計規格文件**（`design-spec.md`，從主文件各步驟錨點連結）：
- 每個步驟的元件 ID、視覺規格（顏色、間距、字體大小）
- 完整文案庫（含 A/B 版本、錯誤訊息、空狀態）
- 工程師實作 checklist

## MODE B：GENERATE — 生成模式

- 這個頁面的**唯一核心目標行動**是什麼？
- 有競品參考或風格偏好嗎？

### 輸出格式

## 通用輸出規範

### 文件層級（所有模式一致）

```
主文件（Markdown）          ← 唯一入口，自給自足，包含所有設計決策
  ├── #錨點 → 互動原型      ← HTML/React，補充「需要互動才能理解」的部分
  └── §連結 → design-spec  ← 純實作細節，設計決策仍在主文件
```

**Markdown 主文件必須「自給自足」**——即使不開啟輔助文件，也能完整理解設計邏輯和心理學根據。

### 每次輸出的五個自我檢查

- [ ] 每個設計決策都有說明對應的心理學原則
- [ ] 有量化數據支撐（研究數據 or 預期效果）
- [ ] 給出了明確的優先順序（先做哪個）
- [ ] 每個建議具體到「把 XX 改成 XX」
- [ ] 有主動提供或詢問是否需要 HTML/React 互動原型
